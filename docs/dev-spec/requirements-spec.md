## 1. 專案背景

本專案是一套基於 Line Bot、n8n 自動化流程、網路爬蟲、機器學習與 AI 分析的智能商業與輿情分析服務。主要目標是協助店家收集網路上與自身品牌、分店、服務或產品相關的輿論資料，分析好評與負評，並在偵測到重要負面評論或異常輿情時，即時透過 Line 通知客戶。

系統初期 MVP 以「輿情分析服務」為主，包含客戶透過 Line Bot 提出服務請求、n8n 傳遞參數與觸發爬蟲任務、系統收集貼文與留言資料、分析正負面情緒、整理數據報告、產生改善建議與警示通知。

後續階段將擴充「線上預約智能管理」功能，並進一步結合店家預約資料、顧客行為資料與輿情資料，提供更完整的商業洞察與經營建議。

## 2. 專案目標

本專案的核心目標是讓中小型店家能以低門檻方式使用智能輿情分析服務，不需要自行操作後台，只要透過 Line Bot 提出需求，就能啟動資料收集、分析與通知流程。

系統需要完成三個主要價值。第一，協助店家持續監測網路評論與討論內容。第二，將非結構化評論資料轉換成可分析的指標，例如正負評比例、留言量、互動量、關鍵議題與高風險評論。第三，當出現明顯負評、客訴、服務問題或品牌風險時，能自動產生警示通知與改善建議。

## 3. 使用者角色

| 角色 | 說明 | 主要需求 |
|---|---|---|
| 店家客戶 Client | 使用 Line Bot 申請服務的商家或負責人 | 提出監測需求、接收報告與警示 |
| 系統管理者 Admin | 維護平台、客戶資料、爬蟲任務與資料品質的人員 | 管理客戶、檢查任務狀態、處理錯誤 |
| 資料分析模組 AI/Data Module | 負責分析評論內容與產出洞察的系統模組 | 判斷情緒、分類議題、產出建議 |
| n8n 自動化流程 | 負責接收 Line Bot 參數並觸發後續服務流程 | 傳遞參數、排程任務、觸發通知 |

## 4. MVP 範圍

MVP 階段優先完成輿情分析服務。客戶可以透過 Line Bot 提供店家名稱、平台、關鍵字或監測需求。n8n 接收參數後建立服務任務，系統根據任務執行爬蟲，收集貼文與留言資料，並將資料寫入資料庫。分析模組會針對資料進行情緒分析、好負評統計、關鍵議題整理與警示判斷，最後透過 Line Bot 回傳結果給客戶。

MVP 不優先完成完整預約系統、會員後台、金流功能、複雜權限管理與正式商業化儀表板。這些功能會列入第二階段或後續進階規劃。

## 5. 功能需求

| 編號 | 功能名稱 | 功能描述 | 優先級 |
|---|---|---|---|
| FR-001 | Line Bot 接收服務請求 | 使用者可以透過 Line Bot 輸入店家名稱、平台、關鍵字或監測需求 | Must |
| FR-002 | n8n 參數傳遞 | n8n 需能接收 Line Bot 傳入資料，並呼叫後端 API 建立服務任務 | Must |
| FR-003 | 客戶資料管理 | 系統需保存客戶基本資料、Line user ID、店家名稱與分店資料 | Must |
| FR-004 | 服務任務建立 | 系統需根據客戶請求建立 service_tasks | Must |
| FR-005 | 爬蟲任務建立 | 系統需根據服務任務建立 crawl_jobs，並記錄平台、關鍵字、狀態與錯誤訊息 | Must |
| FR-006 | 貼文資料收集 | 系統需保存 crawl_posts，包含標題、作者、內容、發布時間、連結與收集時間 | Must |
| FR-007 | 留言資料收集 | 系統需保存 comments，包含留言作者、內容、發布時間與收集時間 | Must |
| FR-008 | 互動數據保存 | 系統需保存貼文與留言的互動數據，例如按讚數、留言數、分享數、觀看數、反應數 | Should |
| FR-009 | 情緒分析 | 系統需判斷評論或貼文內容偏正面、負面或中立 | Must |
| FR-010 | 輿情警示通知 | 當偵測到負評、抱怨、客訴或高風險內容時，系統需透過 Line 通知客戶 | Must |
| FR-011 | 分析摘要 | 系統需彙整好評、負評、常見問題與改善建議 | Must |
| FR-012 | 預約管理 | 使用者可透過 Line Bot 進行預約查詢、建立或管理 | Future |
| FR-013 | 商業洞察分析 | 系統可結合預約資料與輿情資料，分析客流、服務問題與經營建議 | Future |

## 6. 非功能需求

| 類型 | 需求描述 |
|---|---|
| 可維護性 | 專案需清楚區分 Line Bot、n8n、API、Crawler、Analysis、Database 等模組 |
| 可擴充性 | 未來需支援更多社群平台、評論來源與商業分析功能 |
| 資料一致性 | 貼文與留言需避免重複寫入，需設計 dedupe key 或唯一判斷邏輯 |
| 錯誤追蹤 | 每次爬蟲任務需記錄成功、失敗、錯誤訊息與執行時間 |
| 資料彈性 | 不同平台欄位不完全一致，需使用 extra_data JSON 保存平台特有資料 |
| 安全性 | Line user ID、客戶資料與 API Token 不得寫死在程式碼中 |
| 效能 | MVP 階段需可支援多個客戶、多平台、多任務查詢 |
| 可測試性 | API、爬蟲、資料庫寫入、AI 分析與通知流程需可個別測試 |

## 7. 驗收標準

MVP 完成時，系統應該可以讓使用者透過 Line Bot 提出店家輿情監測需求。n8n 可以成功接收參數並建立任務。後端可以建立 service_tasks 與 crawl_jobs。爬蟲可以收集至少一個平台的貼文與留言資料。資料庫可以保存貼文、留言與互動數據。系統可以分析正負面評論，並在出現明顯負評時透過 Line Bot 發送警示通知。

## 8. 專案限制與風險

| 風險 | 說明 | 對策 |
|---|---|---|
| 平台反爬蟲限制 | Dcard、PTT、Google Maps 等平台可能限制爬蟲 | 優先使用公開資料、控制頻率、保留錯誤紀錄 |
| 評論語意難判斷 | 反諷、雙關、隱晦抱怨可能導致情緒分析錯誤 | MVP 先用簡化分類，後續導入 LLM 或人工校正資料 |
| Token 成本過高 | 若每則評論都丟給 LLM，成本會快速增加 | 使用規則、ML 模型、embedding 初篩，再對高風險內容使用 LLM |
| 資料重複 | 多次爬蟲可能收集到相同貼文或留言 | 設計 dedupe key 或平台來源唯一識別邏輯 |
| 系統範圍過大 | 同時做輿情分析與預約系統可能導致 MVP 延遲 | MVP 先完成輿情分析，預約功能列為第二階段 |
